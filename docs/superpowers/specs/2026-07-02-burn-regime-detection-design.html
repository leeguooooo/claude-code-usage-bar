烧速政权检测(Burn-Rate Regime Detection)

claude-statusbar v3.19.0 · 2026-07-02 · 决策依据:46 窗口/1057 时点回测 + 用户约束「快速反应,宁可跳」「预测只谈订阅额度,$ 不进模型」

问题

→NN% 预测的近期速率来自 30/60 分钟回看窗。用户临时切换模型(Sonnet↔Fable 烧速阶跃)或并行 fleet 组成变化(新 session 加入)时,回看窗横跨新旧两个「烧速政权」,把阶跃平均掉——切换后 30–60 分钟预测都在追旧政权。回测证实重度窗口中段低估源于此类爬升。

明确不做:① $ 成本速率信号(产品只谈订阅额度);② 模型烧速权重表(Fable 权重纯猜,多 session 并行无法归因);③ Fable 专属周额度桶(stdin 的 seven_day 只有 All-models 一个数,上游数据边界);④ session 离场检测(无渲染即无事件,且漏检方向是安全侧高估)。

方案

在账号级共享 store(rate_latest.<acct>.json)记政权边界时间戳;速率估计的回看窗被边界截断,边界后放宽最小样本跨度以尽快「跳」上新速率;display 平滑在边界处重置一次,不磨掉跳变。

数据与检测(predict.reconcile_account,仅记录型调用)

项内容
sessions 表扩展v3.18.1 的 sessions[sid] 增加 model 字段 = 该 session 最近一次 stdin 的 model.id(core.main 传入;缺失时保留旧值,不清空)
新 store 键regime: {changed_at, reason, model};GC 不触碰,后来的 bump 覆盖
触发 ①已知 session 的 model.id 变化(reason=model-switch)
触发 ②新 session 首见,且其模型不在活跃 fleet(changed_at 距今 ≤ FLEET_ACTIVE_S=900s 的 sessions 的 model 集合)中(reason=fleet-join)。同模型加入不触发,避免频繁开窗把回看窗永久打短
不触发record=False 回声调用;model 缺失;首个 session(无历史可截断)

生效(速率估计与显示)

位置行为
_rate_from_samples(…, since)有效 cutoff = max(now − lookback, since);当截断生效时最小跨度从 300s 放宽到 REGIME_MIN_SPAN_S=120s。样本仍不足 → 返回 None,混合退化为平均+时段先验(盲区 ~2–5 分钟,可接受)
project_5h / project_7d新增 since 形参,透传给 30m/60m 回看与 7d 动量速率
projection()每次调用读一次 regime_changed_at()(新 helper,读共享 store,失败返回 None),传入 _projection_for_window
display 平滑_projection_for_window 中若 previous.updated_at < since → 丢弃 previous,该次渲染直接跳到 raw,后续从新起点继续平滑

边界与风险

情形行为
反复 A/B 切换反复 bump,回看窗持续被打短,预测更抖——「宁可跳」的自选代价
速率样本只截断速率估计,样本本身保留(时段先验学习不受影响)
历史回测存量样本未记录模型,政权检测无法离线验证;靠单元测试 + 实机观察
旧版共存旧版本读 store 忽略 regime 键;新版本读无该键的 store → since=None,行为同 v3.18.1
